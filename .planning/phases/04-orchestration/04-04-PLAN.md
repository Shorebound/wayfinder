template<typename T>
concept ServiceProvider = requires(T provider) {
    { provider.template Get<int>() } -> std::same_as<int&>;
    { provider.template TryGet<int>() } -> std::same_as<int*>;
};

class StandaloneServiceProvider {
    template<typename T> void Register(T& service);
    template<typename T> auto Get() -> T&;
    template<typename T> auto TryGet() -> T*;
};

    Class `Simulation` derives from `StateSubsystem`:

    **Public API:**
    - `Simulation() = default;` (default constructor, world created inline)
    - `~Simulation() override;` (defined in cpp to handle Scene unique_ptr)
    - `[[nodiscard]] auto Initialise(EngineContext& context) -> Result<void> override;` - Base subsystem init. Minimal setup -- just logs. The flecs world is created in the constructor (flecs::world has a default constructor). Store a reference to EngineContext for later use if needed.
    - `void Shutdown() override;` - Unload current scene, reset world.
    - `void Update(float deltaTime);` - Tick the flecs world: `m_world.progress(deltaTime);`
    - `[[nodiscard]] auto GetWorld() -> flecs::world&;`
    - `[[nodiscard]] auto GetWorld() const -> const flecs::world&;`
    - `[[nodiscard]] auto GetCurrentScene() -> Scene*;`
    - `[[nodiscard]] auto GetCurrentScene() const -> const Scene*;`
    - `void LoadScene(std::string_view scenePath);` - @prototype stub that logs a warning. Full scene loading depends on AssetService which is not yet integrated as a subsystem. Creates a minimal Scene object or sets m_currentScene.
    - `void UnloadCurrentScene();` - Reset m_currentScene.

    **Private members:**
    - `flecs::world m_world;`
    - `std::unique_ptr<Scene> m_currentScene;`

    Note per D-09: Simulation is THIN. It owns only the flecs world and scene. Physics, audio, etc. are separate state subsystems. Per D-11: Simulation IS a StateSubsystem, accessed via `context.GetStateSubsystem<Simulation>()`.

    Per D-12 (SIM-06): Sub-states are state-internal. The ApplicationState creates its own `StateMachine<TStateId>` in OnEnter. The engine provides the tool (StateMachine from Phase 1) but does NOT manage sub-state usage. No sub-state registration API on Simulation. This requirement is satisfied by the existing StateMachine<TStateId> template.

    Per SIM-07: ActiveGameState singleton is updated by game code (e.g., in a GameplayState's sub-state machine OnTransition callback) via `world.set<ActiveGameState>({...})`. Simulation provides GetWorld() for this. No engine-managed mechanism needed.
