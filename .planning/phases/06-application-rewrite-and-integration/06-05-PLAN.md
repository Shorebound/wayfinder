    **main() function (D-08):**
    ```cpp
    int main(int argc, char* argv[])
    {
        const Wayfinder::Application::CommandLineArgs args{.Count = argc, .Args = argv};
        Wayfinder::Application app(args);

        // Platform + rendering plugins
        app.AddPlugin<Wayfinder::SDLPlatformPlugins>();
        app.AddPlugin<Wayfinder::SDLRenderDevicePlugin>();
        app.AddPlugin<Wayfinder::EngineRenderPlugin>();

        // Game plugins
        app.AddPlugin<JourneyPlugin>();

        app.Run();
        return 0;
    }
    ```

    **JourneyPlugin (replaces old JourneyGame class):**
    Convert from `Plugins::Plugin` to `IPlugin`. Instead of `registry.AddPlugin<X>()`, use `builder.AddPlugin<X>()`.
    ```cpp
    class JourneyPlugin : public Wayfinder::IPlugin
    {
    public:
        void Build(Wayfinder::AppBuilder& builder) override
        {
            // Register Simulation as state subsystem (available when GameplayState enters)
            builder.RegisterStateSubsystem<Wayfinder::Simulation>();

            // Register GameplayState as the initial application state
            builder.AddState<Wayfinder::GameplayState>();
            // If there were other states: builder.AddTransition<GameplayState, PausedState>();

            // Physics via AppBuilder per D-09
            builder.AddPlugin<Wayfinder::Physics::PhysicsPlugin>();

            // Game-specific plugins that register ECS components/systems
            builder.AddPlugin<HealthPlugin>();
            builder.AddPlugin<GameplayPlugin>();
            builder.AddPlugin<TagDemoPlugin>();
        }

        [[nodiscard]] auto Describe() const -> Wayfinder::PluginDescriptor override
        {
            return {.Name = "JourneyPlugin"};
        }
    };
    ```

    **HealthPlugin conversion:**
    Convert from `Plugins::Plugin` + `PluginRegistry` to `IPlugin` + `AppBuilder`.

    The v1 HealthPlugin registered ECS components via `registry.RegisterComponent()`. In v2, the AppBuilder doesn't have a direct RegisterComponent API -- this was a PluginRegistry feature.

    **Approach for ECS component/system registration in v2:**
    Read the actual codebase to determine the v2 pattern. Options:
    1. If AppBuilder has been extended with RegisterComponent/RegisterSystem, use that
    2. If Simulation exposes a registration API, use lifecycle hooks
    3. If PluginRegistry lives on as a game-level concept passed through AppDescriptor, use it
    4. If SceneWorldBootstrap has a registration mechanism, use that

    **Most likely v2 pattern (read code to confirm):**
    The v1 PluginRegistry::RegisterComponent was about scene serialisation (key -> register/apply/serialise/validate functions). This is separate from the v2 subsystem/state infrastructure. Check:
    - Does AppBuilder::Finalise() still accept component descriptors?
    - Does a ComponentRegistry or RuntimeComponentRegistry survive in v2?
    - Does SceneWorldBootstrap accept custom component descriptors?

    If the component registration system survives as its own thing (RuntimeComponentRegistry or similar), then:
    ```cpp
    class HealthPlugin : public Wayfinder::IPlugin
    {
    public:
        void Build(Wayfinder::AppBuilder& builder) override
        {
            // Register component descriptor via whatever v2 mechanism exists
            // This may need a ComponentRegistryPlugin or lifecycle hook
        }
    };
    ```

    If no v2 mechanism exists yet for component registration, mark with `@prototype` and a TODO noting it needs a proper registration path. The minimal viable approach: keep PluginRegistry for now and have JourneyPlugin access it, or inline the registration into GameplayState::OnEnter.

    **GameplayPlugin conversion:**
    The v1 API registered states by name ("Playing", "Paused") and conditioned ECS systems. In v2:
    - Named game states (Playing/Paused) are sub-states WITHIN GameplayState, managed by GameplayState's internal state machine, NOT by the ApplicationStateMachine
    - ECS systems registered via whatever mechanism HealthPlugin uses
    - Same approach as HealthPlugin for the registration pattern

    **TagDemoPlugin conversion:**
    Tags are managed by TagRegistry. Check if TagRegistry survives in v2 and how it's accessed. Likely through EngineContext or a dedicated subsystem.

    **General conversion rules:**
    - Replace `Plugins::Plugin` base with `IPlugin`
    - Replace `Plugins::PluginRegistry&` param with `AppBuilder&` param
    - Replace `registry.AddPlugin<X>()` with `builder.AddPlugin<X>()`
    - Replace `registry.RegisterState(...)` with internal sub-state pattern
    - Replace `registry.RegisterSystem(...)` with v2 ECS registration
    - Replace `registry.RegisterComponent(...)` with v2 component registration
    - Replace `registry.RegisterTag(...)` with v2 tag registration
